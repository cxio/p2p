# 项目设计

本项目旨在设计一个精简的去中心化P2P节点发现和连接系统，提供一个P2P世界的基础网络（简称**基网**）。

任意P2P应用节点都可以连入基网，直接获得基网的节点发现功能。它们通常有自己的网络（称为**子网**），子网可以直接复用基网库中的网络基础功能（连接、分享、节点池维护、通用消息查找等）。

这样，应用子网只需要专注于自身特有的功能和服务，开发工作会更简单和轻松。


## 技术栈

- **UDP**：采用 QUIC 作为主要的通讯方式。
- **TCP**：适用于UDP受限的网络环境，采用 HTTP/2 协议，可与UDP方式同时运行（由配置启用）。
- **加密**：采用 TLS v1.3 安全传输协议，支持后量子加密算法（如 X25519MLKEM768）。
- **消息格式**：采用 Protobuf 进行消息序列化和反序列化。
- **语言**：使用 Go 语言实现，充分利用并发能力。



## 节点信息

- 版本：版本号（int）。
- 名识：应用的名称或其它标识，同类节点相同。
- 协议：`quic`, `http2`，通讯协议。QUIC为主，HTTP/2为辅。也可配置为单独支持。
- 地址：公网IP:Port，可以有多个。
- SPKI：证书的公钥信息指纹，即 sha256(SPKI)。
- NATT：NAT类型（`Public` | `FullC` | `RC` | `P-RC` | `Sym`），可选（仅UDP需要）。
- Extra：额外信息，由有需求的应用自行定义。

TCP 采用 HTTP/2 协议以弱化特征，避免处于审查严格环境下的窘境。

基网没有应用功能，名识为一个空字符串（或某固定值）。应用子网的名识由应用自己定义，通常就是应用的名称，或名称的哈希序列。当然，名识实际上也可以是一个随机序列，但应当有确定性。



## 节点搜寻

这是对通用**广播查找**的一种具体实现（只对名识作判断）。

应用客户端向基网发出查找请求，请求包含*目标ID*和一个*跳数*计数。目标ID即为客户端自己的名识。

基网中的节点收到请求后，如果目标ID与自己的名识相同，则回应请求，否则将跳数加一后转播。上级节点收到回应后，会将回应的结果*原路回传*……

这样，正向的搜寻请求逐级扩散，而回应的消息则逐级收敛。没有消息风暴的问题。

> **注意：**
> 节点匹配回应后，即不再转播搜寻请求（适时终止）。
> 如果收到重复的请求，或超出一定时间范围的请求，也不会再转播。


### 搜寻请求

应用客户端通常需要搜寻同类节点，然后才能进入其网络。不过搜寻是自由的，节点可以搜寻任何关联的类型，比如一个NAT内网节点会先搜寻STUN服务节点，然后连入其STUN服务子网，从而获得NAT探测和打洞服务。

搜寻请求包称为*询问包*，有如下内容。

> **注：**
> 下面的伪代码中：`[n]` 表示Bit位数，`(n)` 表示字节数。

```go
(4)     Ver：版本号。
(8)     ID： 标识本次询问，用于回复包中对应其源请求。
(8)     Begin：起始时间戳（Unix时间，秒级）。
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
[4]     跳数累计：上限值15，可以是非零起跳。
[4]     公钥算法：如 X25519。
(32)    公钥序列：用于回复包中联系信息的加密。
(n)     目标ID：
        对于节点搜寻来说，即为名识。
        对于其它用途，则为广播查找的消息ID（字节序列，由应用自行定义）。
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
(n)     附加数据：
        可选，作为应用的扩展区。
        为避免请求包过大，可能会有严格的限制（如 <256字节）。
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
```

**计数器**有一个固定的上限值（`<16`），超过后即不再转播，避免请求在网络中无限循环。客户端可以通过设置一个非零起跳值来控制请求的传播范围。


### 回应消息

如果节点匹配传入的目标名识（或匹配广播查找的目标），它需要回应该请求。如果必要，可以附带扩展内容（Extra）。

回应消息按原始广播路径回传，有如下信息。

**回复包**结构：

```go
(4)     Ver：版本号。
(8)     询问ID：原询问包中的ID，用于上级对应请求源。
        Reply：应答节点回复的时间戳。
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
(32)    公钥序列：
        应答者的公钥，用于询问者构造共享密钥。算法与询问包相同。
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
注：以下信息已用共享密钥加密
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
(1)     跳数累计：
        询问包到达应答节点时的跳数，可用于距离评估。
        加密后传递防止探知和篡改，维护连接的随机性，避免物理分割攻击。
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
(n)     目标ID：
        与询问包内的同字段值相同。
(n)     回复数据：
        对于节点搜寻来说，就是应答节点自己的联系信息。
        对于其它用途，则为广播查找的消息回应数据（字节序列，由应用自行定义）。
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
```


### 回传策略

上级节点会收到多个回应包，它们来自不同的下级节点。回传遵循如下规则：

- 每个节点最多向上回传8个包（多出的会被上级节点丢弃）。
- 回复包会先进入一个暂存区，等待一个延时，之后随机选取最多8个包向上回传。
- 如果暂存区中的包数量不足8个，则全部回传。然后开启下一轮延时收集。

> **注：**
> 每轮延时从当期第一个包到达后开始计时。

这样，如果上级节点连接了8个下级节点，则最多可获得64个回复。这对于寻找初始接入点来说应该已经足够。



## 安全连接

由于P2P网络的节点间不需要信任对方，因此连接仅需采用安全连接（TLS），避免第三方的窥探和分析即可。

这里采用自签名证书创建安全连接，证书的验证采用 SPKI 指纹（sha256(SPKI)）核对。

> **提示：**
> 节点证书的 SPKI 指纹会作为节点信息之一，在节点搜寻回应中携带。
> 在应用子网中，包含 SPKI 指纹的节点信息也会在节点间分享、扩散（与基网的逻辑一样）。



## 实现参考

### 候选池

暂存可直接连入的 `Public` 和 `FullC` 类节点信息，当准备分享时，从中随机抽取部分节点（比如15个）。

候选池节点也是基网主动连出时的目标，当进行连接切换时就会从中随机选取一个或几个节点。而当连接建立后，节点也会从对方获得分享（可能有一个随机延时，以避免流量特征）。

收到的分享节点会测试其可达性，但出于“毒IP”的设计，只会测试部分成员（局部模糊策略）。

> **注意：**
> 除 `Public` 和 `FullC` 类型之外的节点不能对外分享，因此不进入候选池。
> 候选池中只有节点的连系信息，并没有当前连接的实例（测试后即已断开）。

候选池大小有一个上限值（可配置），避免低性能机器的资源消耗过多。


### 组网池

即当前组网节点的*连接池*，维持基网的持续运行。组网池分为两个部分：**连入**和**连出**。

主动连出的节点数上限通常比连入节点数上限低，但连出的都是 `Public/FullC` 类节点，而连入的则是任意NAT类型的节点。

组网池中的节点连接需要维护，比如维持适当的规模、不定时切换节点和接受请求分享候选池中的节点。

> **注意：**
> 组网池不是候选池，候选池中只有节点信息而非实际连接。


#### 转发策略

在节点对消息的转播中，并不是全量转发，而只是随机选取一部分节点转播（可配置）。这里有一个策略：

- 连入与连出的节点分别取一半（按配置值）。
- 先取连入的节点，如果数量不足，则剩余部分全部从连出节点池中获取。

这可以自适应连入节点数不足的情况，然后让连出节点占优（获得更多的分配）。


#### 更新策略

组网池中的节点连接需要定期更新，以维持网络的健康和稳定。更新策略包括：

- 每2~3分钟随机抽取一个连接用于更新（新连接建立后会断开该旧连接）。
- 会检查时间戳，抽取的连接至少需超过10分钟才会被更新，否则另选其他连接，避免不稳定性。
- 若为8个连出，大概16~24分钟会完整更新一次。这有一定的模糊性，包容。


### 初始节点植入

初始节点可以由应用内置或用户配置导入。如果两种方式都没有，则采用暴力发现的方式遍历IPv4地址进行搜寻。

在网络发展的初期，基网的规模不大，在App中植入一些已知的基网节点是必要的。


### 被动分享

节点不应当主动分享自己候选池中的信息，而应该在对方请求后才提供。这更简单和合理。



## 名词解释

### 基网纯节点

仅实现基网本身的功能，没有任何其它用途的节点，称为基网的**纯节点**。

它们是基网初期扩展的基础（通常成本低廉），同时也可能是未来远期整网的融洽成份（或许会有某些必要的进一步工作）。


### 基网兼节点

连入基网的应用节点，它们是基网的重要组成部分，不过有*兼职*的观感。

兼职基网并不会占用多少资源。所以一个应用节点如果从基网获得了需要的入口（理论上就可以脱离基网了），但还是希望继续保持连接，这可以极大地扩展基网规模，从而让基网真正普及（甚至秒连）。

兼节点不可或缺！它们是基网百万级以上规模的主要力量。


### 自由市场

所有参与基网连接的节点的集合（基网纯节点+任意子网节点），统称为节点的*自由市场*。

这只是一种逻辑上的称谓，便于表达，不是另有一个独立实体。基网的逻辑不包含应用子网其自身的功能部分。

自由市场的持续增长可能带来真正的去中心化（完全没有中心化机制），使得一个很冷门或个性化的应用，都不用担心无法成网。理论上，即便全网只有2个某应用的节点，它们都可以找到彼此。



## 关于服务子网

服务子网是一些公共的服务，比如 STUN 探测和打洞服务、数据中转服务等。它们由单独的项目创建，这里无需涉及。



## 配置条目参考

```hjson
{
    // ========== 基本配置 ==========
    server_uport: 7788,             // QUIC 服务器端口
    server_tport: 443,              // HTTP/2 服务器端口
    enable_http2: true,             // 是否启用HTTP/2
    http2_cleartext: false,         // HTTP/2 是否明文

    // ========== 节点池 ==========
    pools: {
        shortlist_size: 100,        // 候选池大小
        max_outgoing_peers: 10,     // 连出节点数上限（组网池）
        max_incoming_peers: 50,     // 连入节点数上限（组网池）
        share_peers: 15,            // 分享节点数（含毒IP）
        connection_timeout: 10,     // 连接超时(秒)
        test_ratio: 0.65,           // 可达性测试比例
    },

    // ========== 节点搜寻 ==========
    query: {
        reply_delay: 500,           // 回复收集延时(ms)
        max_relay_peers: 8,         // 转播&回传最多节点数
        cache_enabled: true,        // 启用缓存
        cache_ttl: 300,             // 缓存有效期(秒)
        cache_max_entries: 100,     // 最大缓存条目数
    },

    // ========== 毒IP策略 ==========
    poison_ip: {
        enabled: true,
        total_ratio: 0.30,              // 总占比30%
        confirmed_ratio: 0.15,          // 确认的大平台IP
        likely_ratio: 0.10,             // 可能的大平台IP
        experimental_ratio: 0.05,       // 实验性IP
        dynamic_adjust: true,           // 动态调整
    },

    // ========== 节点评估 ==========
    evaluation: {
        enabled: true,
        update_interval: 300,           // 评估更新间隔(秒)
        min_samples: 10,                // 最少样本数

        // 权重配置
        weights: {
            uptime: 0.25,
            latency: 0.20,
            bandwidth: 0.15,
            reliability: 0.20,
            share_quality: 0.10,
            response_rate: 0.10,
        },
    },

    // ========== 节点发现（暴力） ==========
    discovery: {
        mode: "auto",                    // auto | aggressive | passive | manual
        timeout: 300,                    // 发现超时(秒)
        concurrent: 1000,                // 并发探测数
        cloud_first: true,               // 优先云服务商IP段
        use_history: true,               // 使用历史记录
        history_file: "~/.p2p/history.json",
        history_ttl: 2592000,            // 历史有效期（30天）
    }
}
```
