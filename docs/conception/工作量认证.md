# 工作量认证

节点之间的连接需要先执行一个工作量运算，以设置一种开放式认证。防止某些攻击或探测，维护网络的良好运行。

工作量算法基于非对称代价的 `Equi-X` PoW 哈希，连接方需要付出更多的计算，而接受/验证方计算成本较低。


## 基本算法

```go
// @AppSeed 32字节客户端固定种子，App升级时可能更新。审查应变因子。
// @ServAddr 服务端地址，IP:Port格式字符串
// @ClientIP 客户端公网IP地址字符串（无端口）
// @Timestamp 当前时间戳，容许 `±5分钟` 的误差
// @Rand16 16字节随机数，CH.random 的后16字节
challengeSeed := SHA256( AppSeed || ServAddr || ClientIP || Timestamp || Rand16 || ECH.SNI )

// @appNonce: uint64 随机数，客户端内置，升级时通常会更新。审查应变因子。
// 客户端工作量运算：appNonce 作为内部 nonce 不变，通过调整 Rand16 来改变 challengeSeed 求解。
// @solution 返回的8个哈希索引（每索引2字节），共16字节。
solution, err := equix.Solve(challengeSeed, appNonce)

// 首包即提供认证信息。
CH.random = solution || Rand16
```


### 安全性

内在确定性的 solution 会削减 CH.random 信息熵，但 solution 本身依然有随机性。

为避免中间审查者通过验证 CH.random 特征来发现这是一个工作量认证的连接请求，客户端可内置多个 appNonce（任一符合皆可），这也是App升级后的自然状态。

对于中间审查者来说，可核查的只有 CH.random 一个目标，而且外观随机，与任何其它 TLS 连接请求无异。在海量的 TLS 连接请求中，这种抽查的费效比应该很低，尤其是在 P2P 这种自由连接且动态变化频繁的场景下，连接请求无法被即时阻断，对 P2P 节点就是一种价值。


## 响应分级

发起连接一端的初始 ClientHello 中即包含工作量证明，服务端进行验证。有如下三种情况：

0. 非法连接：无工作量证明或验证未通过，不予理会，或伪造其它响应（应对审查探测，视审查环境未来扩展……）。
1. 上线协助：工作量证明验证通过，但 ClientHello 为明文。此为上线协助模式，双方仅有一次性交互（ServerHello 回复）。
2. 正常连接：同上合法的工作量证明通过，同时采用了 ECH 加密。此即正常连接，工作进入下一步……。


### 上线协助

此时客户端发送的 ClientHello 消息包为明文，服务端验证工作量证明后，返回的 ServerHello 包中携带自签名证书和部分节点信息（协助数据）。

客户端解析获取协助数据，其中的节点信息包含了目标节点的地址（IP:Port）、ECH 公钥和自签名证书的 SPKI 指纹等。


### 正常连接

通过上线协助，客户端获得了部分节点信息，至此即可向目标节点发起正常的 TLS 连接请求。此时 ClientHello 需要加密，且同样需要提供工作量证明。

正常连接时，服务端返回的 ServerHello 包里依然携带有初始分享的节点数据（同协助数据）。
