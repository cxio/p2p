# 工作量认证

节点之间的连接需要先执行一个工作量运算，以防止某些攻击，维护网络的良好状态。


## 基本条件

执行工作量哈希运算的因子包含：

- 应用App固定种子（`AppSeed`）值，32字节随机数。可随App升级更新。
- 服务端地址（`ServAddr`，IP:Port）序列化值，让哈希因子依目标不同而不同。
- 客户端公网IP（`ClientIP`，无端口）。
- 实时时间戳。容许 `±5分钟` 的误差，使得客户端&服务器各自计算时相同。
- ClientHello 消息包随机数（`CH.random`），工作量主体因子。
- ClientHello 加密消息包（ECH）中的伪 `SNI`（随机字符串），主要用于信息熵增。
- 临时随机数（`tmpRandom`），仅在需要拥塞控制时才使用，属于二级认证时。

> **注：**
> 难度限定（哈希结果满足特定需求）会削减 CH.random 信息熵，但基本可以忽略。



## 响应分级

发起连接一端的初始 ClientHello 中即包含工作量证明，服务端验证通过后，即可进行下一步的必要回应。

回应分四种情况：

0. 非法连接：未提供工作量证明的视为非法连接，不予理会。
1. 上线协助：对端提供了*初始工作量*证明，但ClientHello为明文。此为一次性交互的上线协助。
2. 一级认证：同上*初始工作量*验证通过，采用ECH加密，两端建立正常连接。
3. 拥塞控制：二级认证，如果节点负载较大，新连入的节点会被要求进一步的工作量证明。

> **注：**
> 拥塞控制工作量难度与负载压力正相关，是一个运行时动态值。

综合权衡，考虑服务端负载需求和抗攻击能力，哈希算法选用抗 ASIC 的 `HashX`。


### 上线协助

此时客户端发送的 ClientHello 消息包为明文，工作量算法如下：

- 采用4级递进匹配，避免简单验算直接看到结果（特征发现）。
- 客户端找到一个随机序列（32字节），使得4级运算满足4个哈希难度。
- 加入客户端公网IP地址（适当降低友好性），促使用户尽可能采用正常的一级认证（ECH）。

```go
// 目标难度序列：
// - 值越小，难度越大。
// - 配置难度从高到低，以快速排除无效请求。
Diff_0 := [4]int{64, 2048, 8192, 16384}

Odd := 0x517cc1b727220a95       // 常数：强化Bit位熵
Mulx := [4]int{11, 13, 17, 19}  // 4级乘数
```

以上信息内置于应用App，预先固化统一。


#### 客户端难度求值

- 找到一个随机序列（Random），然后作为 CH.random 的值。
- 循环迭代4次，每次取哈希结果的2字节，分别满足目标难度值序列。

```go
var Random [32]byte
for {
    // 时间因子（±5分钟）
    ts := time.Now().Unix()
    Timestamp := ((ts + 300) / 600) * 600
    // 32字节随机序列
    rand.Read(Random[:])

    n := 0
    for i := 0; i < 4; i++ {
        proof := Hash( AppSeed || ServAddr || ClientIP || Timestamp || Random || (Odd * Mulx[i]) )
        // 难度满足：小于等于。
        if BigEndian.Uint16(proof[n:n+2]) > Diff_0[i] {
            break
        }
        n += 2
    }
    if n == 8 {
        break   // 找到满足难度序列
    }
}
```

> **注：**
> 难度对比被分散在4次迭代中，每次2字节值。


#### 服务端验证难度

读取客户端 ClientHello.random 值，同样循环迭代4次，验证哈希结果满足目标难度序列。

```go
ts := time.Now().Unix()
Timestamp := ((ts + 300) / 600) * 600
Random := CH.random

n := 0
for i := 0; i < 4; i++ {
    proof := Hash( AppSeed || ServAddr || ClientIP || Timestamp || Random || (Odd * Mulx[i]) )
    // 难度满足：小于等于。
    if BigEndian.Uint16(proof[n:n+2]) > Diff_0[i] {
        return errors.New("工作量难度未达标")
    }
    n += 2
}
return nil  // 难度符合要求
```

> **实现：**
> 服务器应当暂存一个时间周期的 CH.random 值，避免被用于重放攻击。


### 一级认证

通过上线协助，用户已经获得了一些节点信息，其中包含了节点的地址、ECH公钥和自签名证书的SPKI指纹等基本信息。

客户端向目标节点发起连接请求，ClientHello 已有必要的加密。


#### 难度算法

采用简单的哈希结果前置零位长度比较，零位长度要求内置于应用App统一固化。

```go
// 哈希计算：
// @SPKI 节点信息内容之一，sha256(Cert.SPKI)
// @ECH.SNI 随机字符串（5-25 字节）
// @Timestamp 时间戳，算法同上线协助部分
// @CH.random 客户端发送来的 ClientHello.random
Hash( AppSeed || ServAddr || SPKI || ECH.SNI || Timestamp || CH.random )
```


#### 客户端难度求值

客户端循环迭代，找到一个32字节随机序列，采用如上哈希计算，结果满足App内配置的前置零位长度要求。


#### 服务端验证难度

服务器提取客户端发来的 CH.random，根据上面的规则计算哈希，验证结果满足前置零位目标要求。


### 拥塞控制

如果服务节点负载压力大，则在一级认证通过之后，可以要求新连入的节点执行二级工作量认证。

此时的难度由服务端自行决定，难度值（前置零位长度）会设置在 ServerHello 的加密扩展中（`EncryptedExtensions.ZeroLen = n`）。客户端在进入正常的通讯之前，需要先检查该字段值。

> **注意：**
> 拥塞控制仅针对新连入的请求，客户端需要首先检查该值（必须是首个应用数据包——如果需要）。
> 如果该值未设置或为零，则无二级工作量认证要求，正常进入后续工作。

通过了一级认证的客户端已经与服务器建立了加密连接，二级认证在此连接上传输。


#### 难度算法

算法类似一级认证，但参与计算哈希的因子更简化。

```go
// 哈希计算：
// @Timestamp 时间戳，算法同上线协助部分
// @tmpRandom 一个32字节的随机序列
Hash( AppSeed || ServAddr || Timestamp || tmpRandom )
```

客户端完成工作量后，将 `tmpRandom` 作为首个应用层数据包回传到服务器，接受验证。



## 技术参考

为了实现 ClientHello 消息包定制，以及向 ServerHello.EncryptedExtensions 添加自定义字段，可能采用如下技术方案：

- **客户端**：fork `uTLS`，实现读取服务端 EncryptedExtensions 自定义字段。
- **服务端**：用 OpenSSL 库采用 cgo 辅助实现握手控制和向 EncryptedExtensions 添加自定义字段。

客户端实现详见项目：[wTLS](https://github.com/cxio/wtls) 。
