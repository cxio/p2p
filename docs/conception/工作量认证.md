# 工作量认证

节点之间的连接需要先执行一个工作量运算，以防止某些攻击，维护网络的良好状态。


## 基本条件

执行工作量哈希运算的因子包含：

- 客户端固定种子（`AppSeed`）值，32字节随机数。可随App升级更新。
- 服务端地址（`ServAddr`）IP:Port 序列化值，让哈希因子依目标不同而不同。
- 实时时间戳。容许 `±5分钟` 的误差，使得客户端&服务器各自计算时相同。
- 一个秘密随机数（`nonce`），由服务端猜测，主要用来回避直接特征验证（流量审查）。
- ClientHello 消息包随机数（`CH.random`），工作量主体因子。
- ClientHello 加密消息包（ECH）中的伪 `SNI`（随机字符串），主要用于信息熵增。
- 临时随机数（`tmpRandom`），仅在需要拥塞控制时才使用，属于二级认证时。

> **注：**
> 工作量哈希算法可能采用抗 ASIC 的 `HashX` 或 `Argon2id`。
> 难度限定（哈希结果前置0位）会削减 CH.random 信息熵，但基本可以忽略。

工作量*难度值*与 `AppSeed` 一样，会内置于客户端（硬编码，仅升级时可变）。


## 响应分级

发起连接一端初始的 ClientHello 中即包含工作量证明，使得首包即认证。服务端验证通过后，即可进行下一步的必要回应。回应分四种情况：

0. 未提供工作量证明，视为非法连接，简单不予理会。
1. 上线协助：对端提供了*初始工作量*证明，但ClientHello为明文。一次性交互。
2. 一级认证：同上*初始工作量*验证通过，服务端返回证书，创建正常连接。
3. 二级认证：拥塞控制。如果节点负载较大或连入压力大，新连入的节点会被要求进一步的工作量证明。

> **注：**
> 拥塞控制工作量难度随负载压力增大而提升，动态值。


### 初始工作量

初始工作量认证在客户端发送的首包上完成，算法如下：

```go
// 时间戳：±5分钟范围
ts := time.Now().Unix()
Timestamp := ((ts + 300) / 600) * 600

// 常数：强化Bit位熵
BigOdd := 0x517cc1b727220a95

// 上线协助（非ECH）：
// - 客户端大工作量：nonce取随机值后固定，变化 CH.random 值。
// - 服务器小工作量：在一个预知的小范围内搜寻 nonce 值。
// @nonce 范围：32~64。
proof = Hash( AppSeed || ServAddr || Timestamp || CH.random || (BigOdd * nonce) )

// 正常连接（ECH）：
// 节点信息中已包含证书公钥信息指纹，
// 它与 SNI 一起用于信息熵增（P2P 无需真正的 SNI）。
// - 客户端工作量：ECH.SNI 取随机字符串后，变化 CH.random 值。
// - 服务器：解密提取 ECH.SNI，直接验证。
// @SPKI 节点信息内容之一，sha256(Cert.SPKI)
// @ECH.SNI 随机字符串（5-25 字节）
proof = Hash( AppSeed || ServAddr || Timestamp || SPKI || CH.random || ECH.SNI )
```

#### 客户端行为

客户端需要完成工作量来满足预置的难度（迭代计算proof值）。

1. 获取服务端地址（IP:Port）。
2. 创建 nonce 随机值 `N`（上线协助）；或伪 SNI 随机字符串（正常连接）。
3. 创建一个32字节的空间，用安全随机数填充，计算哈希结果（proof）。反复尝试直到满足目标难度。
4. 将该32字节随机序列设置为 ClientHello.random 字段的值。

客户端没有服务器 ECH 公钥时（上线协助），明文发送 ClientHello，否则发送加密（正常连接）。

> **提示：**
> 客户端可能需要验证服务器返回的ServerHello加密扩展中的 `nonce` 字段值。


#### 服务器验证

服务器接收客户端发来的 ClientHello 消息后，提取其中的 random 值。

如果 ClientHello 为明文包，执行*上线协助*流程：

- 创建一个33个成员的整数数组，填充随机不重复的 `32...64` 数值序列。
- 该数组成员即为 nonce 候选值，顺序迭代取成员值 `X`，计算哈希值，验证难度是否满足要求。

如果最终没有满足难度，则对方为非法连接请求，简单丢弃忽略。

如果 ClientHello 为加密（ECH）封包，则按*正常连接*流程：

- 解密提取 SNI 随机字符串。
- 按相应算法验算客户端工作量是否满足要求。

最后，服务器构造 ServerHello 消息包：

- 在加密扩展（EncryptedExtensions.nodes）中添加首分享节点信息（3~10 条）。
- 如果为上线协助，设置 EncryptedExtensions.nonce = `X`。

加密 ServerHello 中必要的部分（TLS 1.3）发送到客户端。

> **建议：**
> 一级认证要求服务端猜测nonce值（多次哈希），因此可能选用 `HashX` 算法。


#### 附：上线协助的服务端负担说明

若 ClientHello 为明文包，服务端会执行上线协助流程，此时服务器需要无条件循环迭代，尝试验证对端的工作量……这对攻击者来说是一种便利：只要它们发送了信息包，服务端就得多干些活。这有点傻缺！

但如果没有这种隐藏设计，中间审查者就可以很方便地验证这种工作量特征。虽然此时 ClientHello 消息包与浏览器的普通请求流量没有区别（注：会用 uTLS 伪装浏览器指纹），但由于验证非常容易，所以肯定就是一种风险。

这是一种无奈的折衷。通过适当扩大验证的难度，迫使审查者也需要执行工作量操作，而它也只能是对海量的请求进行抽查……这对审查者来说，性价比不高，或许因此而迟疑（至少成本上去了）。

另外，上线协助只是节点最初连网的需求。如果节点保存了连接历史，重新上线就可以跳过该阶段（如果历史有效）。因此需求并不是巨大的量级。服务器可以根据自身负载，适度考虑降低其优先级。


### 拥塞控制

如果服务节点负载压力大，需要拒绝部分连入以缓解状况，则可以要求新连入的节点执行二级工作量认证。

此时的难度由服务端自行决定（正相关：负载大 => 难度高）。难度值会设置在 ServerHello 的加密扩展中（`EncryptedExtensions.R = n`）返回，客户端在确认连接正常之前，应当先检查该值。

```go
// 二级认证：
// 工作量：tmpRandom
proof = Hash( AppSeed || ServAddr || Timestamp || tmpRandom )
```

此时双方已可正常传递消息（加密连接），客户端完成工作量后，将 `tmpRandom` 作为首个应用层数据包回传到服务器，接受验证。

> **建议：**
> 二级认证时服务器只需一次计算验证，可能选用 `Argon2id` 算法。



## 技术参考

为了实现 ClientHello 消息包定制，以及向 ServerHello.EncryptedExtensions 添加自定义字段，可能采用如下技术方案：

- **客户端**：fork `uTLS`，实现读取服务端 EncryptedExtensions 自定义字段。
- **服务端**：用 OpenSSL 库采用 cgo 辅助实现握手控制和向 EncryptedExtensions 添加自定义字段。

客户端实现详见项目：[wTLS](https://github.com/cxio/wtls) 。
