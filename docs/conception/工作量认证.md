# 工作量认证

节点之间的连接需要先执行一个工作量运算，以防止某些攻击，维护网络的良好状态。


## 基本条件

执行工作量哈希运算的因子包含：

- 应用App固定种子（`AppSeed`）值，32字节随机数。可随App升级更新。
- 服务端地址（`ServAddr`，IP:Port）序列化值，让哈希因子依目标不同而不同。
- 客户端公网IP（`ClientIP`，无端口）。
- 实时时间戳。容许 `±5分钟` 的误差，使得客户端&服务器各自计算时相同。
- ClientHello 消息包随机数（`CH.random`），工作量主体因子。
- ClientHello 加密消息包（ECH）中的伪 `SNI`（随机字符串），主要用于信息熵增。
- 临时随机数（`tmpRandom`），仅在需要拥塞控制时才使用，属于二级认证时。

> **注：**
> 工作量哈希算法采用抗 ASIC 的 `HashX` 或 `Argon2id`。
> 难度限定（哈希结果满足特定需求）会削减 CH.random 信息熵，但基本可以忽略。



## 响应分级

发起连接一端的初始 ClientHello 中即包含工作量证明，使得首包即认证。服务端验证通过后，即可进行下一步的必要回应。回应分四种情况：

0. 非法连接：未提供工作量证明的视为非法连接，不予理会。
1. 上线协助：对端提供了*初始工作量*证明，但ClientHello为明文。此为一次性交互的上线协助。
2. 一级认证：同上*初始工作量*验证通过，两端建立正常的连接。
3. 拥塞控制：二级认证，如果节点负载较大，新连入的节点会被要求进一步的工作量证明。

> **注：**
> 拥塞控制工作量难度与负载压力正相关，是一个运行时动态值。
> 综合权衡服务端负载需求和抗攻击能力，可能选用 `HashX` 算法更合适。


### 上线协助

此时客户端发送的 ClientHello 消息包为明文。算法如下（伪代码）：

```go
// 常数：强化Bit位熵
BigOdd := 0x517cc1b727220a95

// 乘数：
// 固定4个成员，4级进阶匹配。
MultNums := [4]int{11, 13, 17, 19}

// 目标难度序列：
// uint16范围，应用App内置固定配置。此仅示例。
// 难度比较：小于等于。
// - 值越小，难度越大。
// - 配置难度从高到低，以快速排除无效请求。
DiffNums := [4]int{64, 2048, 8192, 16384}

// 客户端难度求值：
// - 找到一个随机序列（CH.random）。
// - 循环迭代4次，哈希结果前8字节（2*4）分别满足一个难度值。
tmpSeed := AppSeed || ServAddr || ClientIP
for {
    // 时间因子（±5分钟）
    ts := time.Now().Unix()
    Timestamp := ((ts + 300) / 600) * 600
    // 32字节随机序列
    CH.random := [32]byte{...}

    for i := 0; i < 4; i++ {
        proof := Hash( tmpSeed || Timestamp || CH.random || (BigOdd * MultNums[i]) )
        n := i*2
        // 整数按大端序解析
        // 难度满足：小于等于。
        if uint16(proof[n:n+2]) > DiffNums[i] {
            break   // 不满足，重来……
        }
        return CH.random    // 找到目标序列
    }
}
```

```go
// 服务端验证难度：
// 循环迭代4次，哈希结果前8字节分别满足难度序列。
// 注：
// 前8字节的2字节取值被隔离在不同的迭代中。
ts := time.Now().Unix()
Timestamp := ((ts + 300) / 600) * 600

srvSeed := AppSeed || ServAddr || ClientIP || Timestamp || CH.random
for i := 0; i < 4; i++ {
    proof := Hash( srvSeed || (BigOdd * MultNums[i]) )
    n := i*2
    if uint16(proof[n:n+2]) > DiffNums[i] {
        return errors.New("工作量难度未达标")
    }
    return nil  // 验证通过
}
```

#### 难度算法设计

- 采用4级递进匹配，避免简单验算直接看到结果（特征明显）。
- 1级难度配置应当较高，以快速排除无效请求（比如攻击者随机设置 CH.random，不做工作量）。
- 1级难度配置也应考虑审查者简单验算即阻断流量的可能性，难度越低，误杀越严重。


#### 附：关于客户端公网IP因子

上线协助的工作量哈希因子包含了客户端的公网IP，这对用户来说，可能有些不便，但这是有意的。

这种阻碍并非致命，人们可以很容易地获知自己节点的公网IP。适当的阻碍可以避免用户太过轻易地使用上线协助，从而尽量减少非ECH请求的数量（缩减规模）。

这种规模的减少，实际上是一种安全增强，它使得审查者抽检的命中率更低，阻断的附带伤害也更大。


### 一级认证

此时客户端已经拥有服务端的 ECH 密钥，客户端发送的 ClientHello 已有必要的加密。

> **注：**
> 此时客户端也会拥有服务器自签名证书的公钥指纹（`sha256(SPKI)`）。

因此难度算法与上线协助不同，可以设计得更为简单直接。

```go
// 客户端难度求值：
// - 找到一个随机序列（CH.random）。
// - 使得哈希结果前置零位长度满足一个难度值。
// @SPKI 节点信息内容之一，sha256(Cert.SPKI)
// @ECH.SNI 随机字符串（5-25 字节）
tmpSeed := AppSeed || ServAddr || SPKI || ECH.SNI
// 目标难度值（App内置）
diffValue := 1 << 48
for {
    // 时间因子（±5分钟）
    ts := time.Now().Unix()
    Timestamp := ((ts + 300) / 600) * 600

    // 32字节随机序列
    CH.random := [32]byte{...}

    proof := Hash( tmpSeed || Timestamp || CH.random )
    if uint64(proof[:8]) > diffValue {
        continue
    }
    return CH.random    // 找到满足目标难度的值
}
```

```go
// 服务器验证难度：直接验算即可。
ts := time.Now().Unix()
Timestamp := ((ts + 300) / 600) * 600
srvSeed := AppSeed || ServAddr || SPKI || ECH.SNI

// 目标难度值（App内置，与客户端相同）
diffValue := 1 << 48

proof := Hash( srvSeed || Timestamp || CH.random )
if uint64(proof[0:8]) > diffValue {
    return errors.New("工作量难度未达标")
}
return nil
```


### 拥塞控制

如果服务节点负载压力大，需要拒绝部分连入以缓解状况，则可以要求新连入的节点执行二级工作量认证。

此时的难度由服务端自行决定（正相关：负载大 => 难度高）。难度值会设置在 ServerHello 的加密扩展中（`EncryptedExtensions.R = n`）返回，客户端在确认连接正常之前，应当先检查该值。

> **注：**
> 拥塞控制仅针对新连入的请求，客户端需要首先检查该值。
> 如果该值未设置或为零，则无二级工作量认证要求，正常进入后续工作。

此时客户端已通过一级工作量认证，与服务端建立了加密连接。

```go
// 首先判断是否要求。
if SH.EncryptedExtensions.R == 0 {
    return
}
// 客户端难度求值：
for {
    ts := time.Now().Unix()
    Timestamp := ((ts + 300) / 600) * 600

    // 32字节随机序列
    tmpRandom := [32]byte{...}

    proof = Hash( AppSeed || ServAddr || Timestamp || tmpRandom )
    if uint64(proof[0:8]) > SH.EncryptedExtensions.R {
        continue
    }
    return tmpRandom    // 找到满足目标难度的值
}
```

客户端完成工作量后，将 `tmpRandom` 作为首个应用层数据包回传到服务器，接受验证。

```go
// 服务端验证难度：直接验算即可
ts := time.Now().Unix()
Timestamp := ((ts + 300) / 600) * 600

// 目标难度：之前设置的值。
diffValue := SH.EncryptedExtensions.R

proof := Hash(AppSeed || ServAddr || Timestamp || tmpRandom)
if uint64(proof:[0:8]) > diffValue {
    return errors.New("二级工作量难度未达标")
}
return nil  // 成功通过
```



## 技术参考

为了实现 ClientHello 消息包定制，以及向 ServerHello.EncryptedExtensions 添加自定义字段，可能采用如下技术方案：

- **客户端**：fork `uTLS`，实现读取服务端 EncryptedExtensions 自定义字段。
- **服务端**：用 OpenSSL 库采用 cgo 辅助实现握手控制和向 EncryptedExtensions 添加自定义字段。

客户端实现详见项目：[wTLS](https://github.com/cxio/wtls) 。
